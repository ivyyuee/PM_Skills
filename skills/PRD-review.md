# PRD 评审

默认使用简体中文。如果用户明确要求英文，或原始文档明显以英文为主，再切换到英文。

除非用户明确要求英文，否则整个评审过程与输出都应保持中文一致，避免中英文标签混用。像“逐轮引导模式”“直接完整评审”“基于现有材料判断”“问题定义”这类表达，默认都使用自然中文。

## Purpose

把一份 PRD 当作“马上要进正式评审会”的文档来审。目标是在会前暴露薄弱假设、缺失要求、不清晰指标、高风险依赖和未决问题，避免正式评审时被连续追问、临场补锅。

这不是为了苛刻找茬，而是为了提升 PRD 的清晰度、完整度和可落地性。

## Key Concepts

### 这是什么

这是一个 PRD 预评审工作流，重点是：
- 确认你到底想解决什么问题
- 判断 PRD 结构是否完整
- 用真实评审视角提前 challenge 文档
- 输出一份可直接回填到 PRD 的修改建议

### 好的输出长什么样

高质量输出应包括：
- 对 PRD 核心的准确复述
- 一份完整性扫描结果
- 最关键的高价值问题
- 按角色归类的风险与缺口
- 一份能直接指导改稿的修订清单

### 好的交互长什么样

默认优先使用**渐进式提问**，而不是首轮就倾倒完整评审结论。

推荐交互节奏：
- 先给用户选择进入方式，而不是替用户默认直接完整输出
- 先给紧凑版判断，再让用户选择先补哪一个最大缺口
- 一次只问一个问题，优先用编号选项缩小范围
- 每得到一个关键信息，就先用 2-4 行帮用户收敛成明确规则或指标口径
- 当某个缺口已经足够清楚时，立刻产出可直接回填 PRD 的文字
- 只有在用户明确要求“一次给我完整结论”时，才使用直接完整输出

目标不是一次性把所有问题问完，而是帮助用户更轻松地把 PRD 一块一块补完整。

### 适用范围

本 skill 适用于**互联网医疗健康类产品**的 PRD 评审，尤其适合以下场景：
- 在线问诊、挂号、随访、处方、医患沟通
- 体检、检验、报告查看、报告解读、健康档案
- 健康管理、疾病管理、用户分层运营、会员服务
- 健康商城、服务预约、采样/寄送/履约链路
- 含 AI 解读、智能问答、风险提示等健康服务功能

如果需求本身不是医疗健康场景，也可以使用本 workflow 的通用评审框架，只需弱化行业特有问题。

### 这不是什么

- 不是从零开始帮你写 PRD 的流程
- 不是逐像素的交互或视觉设计评审
- 不是正式跨职能评审会的替代品
- 不适合只改一句文案或一个小 bug 的场景

### When to Use This

- 你准备把 PRD 发给研发、设计或老板之前
- 你觉得方向差不多对，但文档还不够扎实
- 你想提前找出边界条件、指标漏洞、范围问题
- 你需要的是“能不能上会”的 readiness 检查，而不只是写作建议

### When NOT to Use This

- 你还没有 PRD，只是想从零搭建结构
- 工作内容只是很小的 tweak 或 bug fix
- 没有任何可用上下文，且用户不愿补充材料
- 当前更适合做 discovery、problem framing 或 prioritization，而不是 review

### Facilitation Source of Truth

当你以多轮引导方式运行本工作流时，使用 `workshop-facilitation` 作为交互协议来源。

这意味着：
- 开场先给 session heads-up 和 entry mode
- 每轮只问一个问题
- 显示进度标签
- 只在关键决策点给编号推荐项
- 被打断时先回答问题，再恢复上下文
- 如果用户要单次输出，走直接完整输出流程

### 互联网医疗健康领域视角（按需启用）

如果当前 PRD 属于互联网医疗健康类产品，在通用评审框架之外，额外关注以下问题：
- 用户身份、实名、家庭成员/代管对象、权限边界是否明确
- 健康数据、病历、检验/体检/报告数据的归属与可见范围是否明确
- 是否涉及合规、知情同意、隐私授权、风险告知、医学免责声明
- 是否存在服务履约链路，例如挂号、问诊、采样、寄送、出报告、购药、随访
- 如果涉及 AI 解读或智能建议，是否明确输入、输出、限制说明、兜底与人工校正方式
- 是否考虑埋点、运营指标、漏斗、留存、转化与复盘所需的数据口径

## Application

### Opening

开场先告诉用户大概时长，并**总是先提供进入方式**；即使用户给了完整 PRD 或文件路径，也不要默认直接给出完整结论，除非用户明确说“直接给结论”“不要多轮”。

默认给出三种进入方式：
1. 逐轮引导模式（默认推荐）
2. 直接完整评审
3. 基于现有材料先判断并标注假设

如果用户没有明确选择，优先推荐逐轮引导模式。

如果用户明确要求单次输出，再走直接完整输出流程：
- 先做 PRD 理解摘要
- 再做完整性扫描
- 接着列关键问题
- 最后输出结构化报告

### Progress Labels

多轮模式下显示明确进度，例如：
- `理解确认 Q1/2`
- `完整性扫描 Q1/1`
- `角色评审 Q2/6`
- `收尾 Q1/1`

### Phase 0：获取 PRD

如果用户没有提供完整 PRD，询问他想怎么提供：
1. 给我一个文档路径
2. 直接在这里描述 PRD
3. 粘贴关键章节

如果给了文件路径，先读取文档，再开始评审。

### Phase 1：理解确认

在开始 challenge 之前，先用 2-3 句话复述 PRD 核心，并明确：
- 目标用户是谁
- 核心问题是什么
- 方案大致是什么
- 成功标准或业务结果是什么

然后询问：
1. 理解正确，继续
2. 有偏差，我来补充
3. 你先按当前理解继续

### Phase 2：完整性扫描

对照标准 PRD 结构检查：
- 问题定义
- 目标用户与角色
- 战略背景 / 为什么现在做
- 方案概览
- 成功指标
- 用户故事与需求要求
- 非本期范围
- 依赖与风险
- 待确认问题

如果当前文档是互联网医疗健康类 PRD，再额外检查：
- 是否明确账号、用户身份、家庭成员/代管对象及权限边界
- 是否明确健康数据、报告、服务记录等数据归属与展示范围
- 是否覆盖合规、授权、知情同意、风险提示等前置条件
- 是否写清关键埋点、运营指标、观察周期，以及事件与指标的映射关系
- 涉及 AI 时，是否明确输入、输出、局限、兜底和人工介入方式

每个部分按三档判断：
- 完整
- 部分
- 缺失

然后输出紧凑版健康度仪表盘，并指出最大的薄弱项。

扫描后询问：
1. 继续进入完整角色评审
2. 先只补最关键缺口（推荐）
3. 只审最薄弱的部分

### Phase 2.5：渐进式补缺口（默认推荐）

如果用户选择“先补最关键缺口”或“只审最薄弱的部分”，按以下方式推进：

1. 先指出当前最需要补的一项，例如：成功指标、权限边界、依赖范围、异常场景、埋点口径
2. 只围绕这一项，一次问一个问题
3. 问题优先使用 3-4 个编号选项，帮助用户快速收敛；必要时提供 `Other`
4. 每得到一个答案，先简短复述成明确规则、定义或口径，再进入下一个相关问题
5. 连续 2-5 轮后，如果已经足够清楚，输出一段“可直接回填 PRD”的文本
6. 然后再问用户下一步：继续补下一块 / 进入角色评审 / 结束

常见的渐进式补缺口顺序：
- 先补目标与指标
- 再补权限与边界
- 再补埋点与运营口径
- 最后补依赖、风险与待确认问题

对于互联网医疗健康类 PRD，优先补以下高频缺口：
- 用户身份、实名、家庭成员或代管对象的权限边界
- 健康数据与服务记录的归属、可见范围与状态流转
- 合规、授权、知情同意、风险提示是否前置明确
- 关键漏斗、运营指标、埋点口径是否可观测、可复盘
- AI 输出的可信边界、限制说明与人工兜底路径

注意：不要一上来把所有缺口都展开。让用户先解决一个最关键问题，再进入下一个。

### Phase 3：角色化评审

提供三种模式：
1. 完整角色评审 — 四个角色依次上场，约 5-10 个问题（推荐）
2. 快速角色评审 — 每个角色只问 1 个最关键问题，约 4 个问题
3. 指定角色视角 — 只审某一个角色视角

角色包括：
- 技术负责人（Tech Lead）
- 交互设计师（UX Designer）
- 测试工程师（QA Engineer）
- 业务方 / 老板（Business Stakeholder）

对每个角色：
- 用一句话切换角色
- 一次只问一个问题
- 简短解释为什么这个问题重要
- 对用户回答给出简短点评
- 根据仍未解决的问题继续追问或切换角色

如果当前 PRD 属于互联网医疗健康类产品，优先从以下方向出题：
- 技术：身份权限、数据归属、接口依赖、状态流转、埋点落点、AI/服务链路兜底
- 交互：授权提示、风险告知、异常态、空态、关键转化节点、跨角色/代管视图理解成本
- 测试：权限组合、边界场景、履约异常、授权失败、合规提示、埋点校验
- 业务：目标用户、why now、业务指标、运营可复盘性、范围取舍、合规与转化的平衡

回答选项优先使用编号：
1. 我有答案
2. 好问题，我确实没覆盖到
3. 这个 PRD 里已经写了
4. Other（补充）

### Phase 4：输出评审报告

生成一份结构化报告，至少包括：
- 整体评价
- 通过 / 有条件通过 / 建议重写
- 问题清单表格
- 上会前必须补的内容
- 可优化但非必须项
- 建议回填到哪些 PRD 章节
- 正式评审会前的准备建议

如果是互联网医疗健康类 PRD，报告中优先明确：
- 身份与权限规则是否闭环
- 健康数据、报告、服务记录归属是否明确
- 合规、授权、风险提示是否充分
- 埋点、运营指标、复盘口径是否具备
- AI 或复杂服务链路的兜底机制是否可落地

### Phase 5：收尾动作

询问用户下一步：
1. 保存报告到文件
2. 针对某个问题深入展开
3. 补充信息后重新跑一轮
4. 到这里结束

## Examples

- “帮我看看这个 PRD 能不能拿去给研发评审。”
- “从技术、测试、业务三个角度帮我 challenge 一下这个需求。”
- “不要多轮，直接给我一份 PRD 预评审报告和修改建议。”

## Common Pitfalls

- 一轮里问太多问题，节奏失控
- 重复问 PRD 里已经写清楚的内容
- 提的问题太泛，不贴合文档本身
- 把所有问题都说得一样严重
- 没有把反馈明确映射回 PRD 章节，导致用户不知道怎么改
- 忽略医疗健康场景里的权限、合规、数据归属与风险提示
- 只写业务目标，不写埋点与指标口径，导致上线后无法验证成效
